# Port x3f to Rust

## Context

`x3f` is a ~10K-LOC C/C++ CLI that converts Sigma Foveon X3F raw files to TIFF/DNG/PPM/JPG. It vendors OpenCV, libtiff, libjpeg-turbo, and zlib in [deps/](deps/) and is built with a hand-rolled GNU Make + [sys.mk](sys.mk) toolchain that only targets `osx-universal`, `linux-x86_64`, and Windows mingw. The hot loop ([convert_data in src/x3f_process.c:1725](src/x3f_process.c#L1725)) is single-threaded and scalar. Documentation is ~3 lines of README plus sparse inline comments. Recent commits show the user is actively researching highlight-recovery algorithms ([reconstruct_highlights](src/x3f_process.c#L688), chroma LUT, RepairPix, matrix-pathology gate) — that work must survive the port intact.

Goals:

1. Make the project approachable to outside contributors via clear architecture docs, idiomatic Rust, and a one-command build.
2. Modernize cross-compilation: macOS Intel, macOS Apple Silicon, iOS, iPadOS, Android, Windows, Linux, WebAssembly. Library + bindings only — no reference UI apps.
3. Improve conversion performance (target ≥2× single-image, ≥4× batch) via row-parallel rayon, planar memory layout, and SIMD in the hot loop.

User-confirmed strategy decisions:

- **Incremental FFI port**: build a Rust shell around the existing C, port module-by-module, keep a working binary at every milestone.
- **Denoise via [opencv-mobile](https://github.com/nihui/opencv-mobile) prebuilt static frameworks** (M9-denoise, done 2026-04-27): re-uses the original OpenCV `fastNlMeansDenoising` C++ source; build.rs fetches the pinned 4.13.0 prebuilt for the target triple; binary cost is ~5 MB on iOS / ~19 MB on Android (vs. 50+ MB upstream OpenCV). On `wasm32-unknown-unknown` the build falls back to the no-op `denoise_stub.c` because that target cannot link C++ stdlib. The earlier "drop OpenCV / re-add as pure-Rust" plan is dropped — opencv-mobile's prebuilts cover every PORT-PLAN target except `wasm32-unknown-unknown`, the OpenCL backend was already gone for those targets so the source-strip cost was minimal, and the legacy NLM tuning carries forward unchanged.
- **Library + bindings only** for mobile/WASM (`x3f-core` crate + C ABI for iOS/Android, `wasm-bindgen` for browsers); no reference apps.

## Recommended approach

Cargo workspace with separated concerns. C source moves to `c/` and is compiled by `cc-rs` from a `-sys` crate; safe Rust API lives in `x3f-core`; CLI is a thin Rust wrapper. C internals are replaced one module at a time with parity testing at every step.

```
x3f/
  Cargo.toml                    # workspace
  c/                            # existing src/* moves here, untouched in M0
  crates/
    x3f-sys/                    # cc-rs builds C, bindgen for headers
    x3f-format/                 # X3F container parsing (later: WASM-consumable alone)
    x3f-core/                   # safe Rust API; high-level Reader/Image/Pipeline
    x3f-cli/                    # binary `x3f_extract`
    x3f-ffi-c/                  # C ABI for iOS/Android consumers (cbindgen)
    x3f-wasm/                   # wasm-bindgen surface
  docs/                         # mdbook (architecture, format ref, contributor guide)
  tests/                        # workspace integration + golden corpus
```

### Milestones

**M0 — Workspace & FFI shim (S/M).** Stand up the workspace; `x3f-sys` `build.rs` compiles every C/C++ file in [src/](src/) **except** the denoise files ([x3f_denoise.cpp](src/x3f_denoise.cpp), [x3f_denoise_aniso.cpp](src/x3f_denoise_aniso.cpp), [x3f_denoise_utils.cpp](src/x3f_denoise_utils.cpp)) — that immediately retires the OpenCV dependency. Bindgen the public headers ([x3f_io.h](src/x3f_io.h), [x3f_process.h](src/x3f_process.h), [x3f_output_dng.h](src/x3f_output_dng.h), [x3f_output_tiff.h](src/x3f_output_tiff.h), [x3f_output_ppm.h](src/x3f_output_ppm.h), [x3f_meta.h](src/x3f_meta.h)). `x3f-core` wraps the C API in safe RAII types. `x3f-cli` reimplements [src/x3f_extract.c](src/x3f_extract.c) in `clap`. Replace the `printf` plumbing in [x3f_printf.c](src/x3f_printf.c) with a function-pointer hook so logs route to a Rust callback (essential for mobile/WASM). **Exit:** `cargo run -- input.x3f -tiff out.tiff` produces byte-identical output to the legacy `x3f_extract` for DP2 Merrill, an SD9-class file, and one Quattro file.

**M1 — CI, build modernization, docs scaffold (S).** GitHub Actions matrix on `macos-14` (arm64), `macos-13` (intel), `ubuntu-latest`, `windows-latest`. `cargo test`, `cargo clippy -D warnings`, `cargo fmt --check`. Top-level `ARCHITECTURE.md` walking through the pipeline. Rewrite `README.md` around `cargo install x3f-cli`. Move legacy [Makefile](Makefile)/[sys.mk](sys.mk) under a `legacy/` directory; keep them runnable for one release as a parity reference.

**M2 — Test strategy redesign (M).** The MD5-based [features/consistency.feature](features/consistency.feature) is incompatible with active highlight-algorithm research. Replace with a three-tier scheme:

- *Tier 1* unit tests (Rust, no real X3F files): parsers, Huffman tables, matrix math, gamma LUT.
- *Tier 2* exact-MD5 for stable surfaces only: container metadata dump, JPEG thumbnail extraction, PPM output.
- *Tier 3* perceptual diff (ΔE2000 / per-channel ε) for processed TIFF/DNG, with per-test thresholds and a "max pixels exceeding ΔE>X" gate.

Replace behave/Python with `cargo test` + `insta` for snapshot tests + a `tests/golden.rs` harness for binary outputs. **This milestone must land before any other porting** — without it, port-induced regressions are indistinguishable from intentional algorithm drift.

**M3 — Crate-replaceable code (M).** Eliminate vendored deps and ~3,400 LOC of boring glue, in this order: PPM ([x3f_output_ppm.c](src/x3f_output_ppm.c)) → pure Rust; TIFF ([x3f_output_tiff.c](src/x3f_output_tiff.c)) → `tiff` crate; JPEG thumbnail → byte-blob copy (no decoder needed; the embedded JFIF stream is just `fwrite`'d); CLI args → already done in M0; matrix math ([x3f_matrix.c](src/x3f_matrix.c)) → still C-call from Rust through M3, native port deferred to M6 when the pipeline ports consume it. **DNG ([x3f_output_dng.c](src/x3f_output_dng.c) + [x3f_dngtags.c](src/x3f_dngtags.c)): wrote a minimal Rust IFD writer (~700 LOC across substrate + tags + profiles + MMCR)** rather than depending on `rawler` (LGPL) or the `tiff` crate (no SubIFD support). zlib drops out with the C DNG writer (use `flate2`). `deps/` is *kept* through M3 because [legacy/Makefile](legacy/Makefile) still references it; it goes away with the rest of the legacy build in **M9**.

**M4 — X3F container & metadata parsing (M).** Port [x3f_io.c](src/x3f_io.c) directory walker, header parsing, property lists, EXIF/maker-note extraction, and [x3f_meta.c](src/x3f_meta.c) to Rust. Entropy decoders are now Rust (M5d), so the layout-assert flow inverts: Rust structs the C side reads must match bindgen layouts (we already do this in entropy.rs / quattro.rs / meta.rs). Replace [x3f_print_meta.c](src/x3f_print_meta.c) with a serde-based dump.

End-to-end DNG output is byte-identical to pre-M4a (verified via MD5 on DP2M / DP0Q / SDQH / SD1M).

- **M4d — Section data loaders + CAMF decoders** ([crates/x3f-sys/src/load.rs](crates/x3f-sys/src/load.rs)) — *done.* `x3f_load_data` and the full call graph beneath it ported to Rust: `x3f_load_property_list` (incl. Tier-2 DNG MD5 unchanged on SD1M (`sigma_sd1_merrill_15` `dcaa9929…`), older raw (`_SDI8040` `41a80ce6…`), and Quattro (`_SDI8284` `c2f70f35…`).

**M5 — Entropy decoders (L).** The riskiest single milestone. Port Huffman decode, TRUE decode (Sigma's predictor + Huffman), and Quattro 2×2 expansion ([x3f_process.c:1913](src/x3f_process.c#L1913)) to Rust. Replace `x3f_area16_t`'s `buf`/`data` ownership split with `Vec<u16>` + a `Plane<'a, T>` view type. **Run C and Rust decoders in parallel in CI for ≥1 release** before deleting the C path; fuzz-test against random byte streams. Tier-3 must show zero ΔE — this is exact integer math. **Required corpus coverage**: SD9/SD10, DP\* Merrill, Quattro H, Quattro non-H. If [x3f_test_files/](x3f_test_files/) doesn't cover all of these, that's a blocker, not a discovery.

**M5 phasing (in progress):**

- **M5a — Quattro 2×2 expansion** ([crates/x3f-sys/src/quattro.rs](crates/x3f-sys/src/quattro.rs)) — *done.* Replaces the M0 `exit(2)` stub with a native Rust bicubic upsampler (Catmull-Rom `a=-0.75` to mirror OpenCV `INTER_CUBIC`) + Yis4T BMT/YUV transforms. Tier-3 self-consistency tests pass on DP0Q and SDQH; output is not bit-exact to the legacy OpenCV path but visually equivalent. No SD9/SD10 corpus is currently available — that variant of the older Huffman path will be ported but cannot be sensor-validated until the corpus is augmented (tracked under "open questions").
- **M5b — TRUE entropy decoder** ([crates/x3f-sys/src/entropy.rs](crates/x3f-sys/src/entropy.rs)) — *done.* Native Rust port of `true_decode` and `true_decode_one_color`, including the bit reader and Huffman tree walk. Tier-3 differential test (`true_decode_c_and_rust_byte_identical`) confirms byte-equal DNG output between C and Rust decoders on plain Merrill (TRUE), DP0Q (Quattro layout), and SDQH variants. - **M5c — Huffman + simple decoders** ([crates/x3f-sys/src/entropy.rs](crates/x3f-sys/src/entropy.rs)) — *ported, partially validated.* `huffman_decode` (X530/10BIT raw + THUMB_HUFFMAN), `simple_decode` (uncompressed variant), and `populate_huffman_tree` are ported. No SD9/SD10 corpus (open question 2 below) and no THUMB_HUFFMAN inputs in the current corpus, so these paths are code-reviewed and structurally tested but not sensor-validated. Adding even one SD9/SD10 file would extend the M5b differential test to cover them.
- **M5d — Cutover** — *done.* Two-step:
  - **M5d-α** flipped the C dispatch default to call into Rust and made the legacy C path opt-in via `X3F_C_DECODE=1`.
  - **M5d-β** deleted the C image decoders entirely from [src/x3f_io.c](src/x3f_io.c): `true_decode_one_color`, `huffman_decode_row`, `simple_decode_row`, `huffman_decode`, `simple_decode`, `true_decode`, `get_huffman_diff`, `get_simple_diff`. Rust calls are inlined directly at the loader callsites in `x3f_load_true` / `x3f_load_huffman_compressed` / `x3f_load_huffman_not_compressed`. The bit reader (`bit_state_t`/`set_bit_state`/`get_bit`), `get_true_diff`, and `print_huffman_tree` are kept because the CAMF type-4/type-5 decoders below still use them; CAMF moves to Rust in M4.
  - The differential `true_decode_c_and_rust_byte_identical` test was removed (no C path to differ against). `dng_raw_plane_is_deterministic` and `quattro_dng_raw_plane_is_deterministic` continue to gate non-determinism.
- **M5e — finish-the-x3f_io.c port + DNG batch determinism fix** ([crates/x3f-sys/src/io.rs](crates/x3f-sys/src/io.rs), [crates/x3f-sys/src/process.rs](crates/x3f-sys/src/process.rs), [crates/x3f-core/src/image.rs](crates/x3f-core/src/image.rs)) — *prerequisite step done.* Two changes:
  1. **Cleanup machinery + directory-entry searchers ported.** `cleanup_huffman_tree`, `cleanup_true`, `cleanup_quattro`, `cleanup_huffman`, `free_camf_entry`, `x3f_delete`, `x3f_get_raw`, `x3f_get_thumb_{plain,huffman,jpeg}`, `x3f_get_camf`, `x3f_get_prop`, plus the `legacy_offset` / `auto_legacy_offset` globals all moved out of [src/x3f_io.c](src/x3f_io.c) into [crates/x3f-sys/src/io.rs](crates/x3f-sys/src/io.rs). All buffers freed here were `libc::malloc`/`calloc`/`realloc`-allocated in the Rust loaders (`load.rs`, `io.rs`, `entropy.rs`), so the matching deallocator stays `libc::free`. With this, `src/x3f_io.c` holds no function bodies and is dropped from the cc-rs build entirely. The deeper `Vec<u16>` + `Plane<'a, T>` migration the original M5e bullet called for is now *unblocked* (allocation AND cleanup are both in Rust) but is left to a follow-up — it is structural cleanup with no perf or correctness impact, and the cleanup port closes the immediate goal of "stop compiling `src/x3f_io.c`."
  2. **DNG batch non-determinism fix.** Setting up the M5e port surfaced a pre-existing rare race in M7d batch mode: across 10 runs of the 15-file corpus, `CLIPPED_IMAGE_MERRILL.X3F` produced 3+ distinct MD5s. Root cause: M7d's `apply_highlight_clip_dng` writes the per-image highlight scale to a `thread_local!` cell *before* its `par_chunks_mut` divide-down pass. Inside that nested rayon iterator the calling thread can work-steal another file's `apply_highlight_clip_dng`, whose own set() then clobbers the cell. Two-part fix: (a) move the `DNG_HIGHLIGHT_SCALE.set(global_max)` to the *last* line of `apply_highlight_clip_dng`, after Pass 3 / cleanup / trace-print, so it's the last cell write on the calling thread before return; (b) snapshot the cell into the returned `Image` struct *immediately* after `x3f_get_image` returns in `crates/x3f-core/src/image.rs::Reader::get_image`, before any other FFI call that could re-enter rayon. The DNG writer reads `image.dng_highlight_scale` instead of calling the FFI side-channel later. After the fix: 10 batch runs produce identical MD5s on every corpus file; 3 primary baselines (SD1M / `_SDI8040` / `_SDI8284`) byte-identical for both DNG and TIFF; full 15-file corpus byte-identical to M7c HEAD output (worktree-rebuilt for the comparison).

- **M5e+ — deeper `x3f_area16_t` → `Vec<u16>` + `Plane<'a, T>` migration.** Deferred. With the cleanup port done, the buffer-allocation site (`x3f_load_true`) and the buffer-free site (`x3f_delete`) are both Rust, so the FFI struct `x3f_area16_t.{buf,data}` can be replaced with a `Box<[u16]>` / `Vec<u16>` owner + `Plane<'a, T>` borrow without breaking ABI for the still-bindgen'd `x3f_image_data_t` consumers (the C ABI struct stays as a layout-compat mirror). Scope: refactor allocations in `load.rs`/`entropy.rs` to use Rust ownership, add a `Plane<'a, T>` view type for new internal APIs, leave the FFI struct unchanged. No perf or correctness impact — purely structural cleanup. Sequence after M9.

**M6 — Processing pipeline port (L).** Port [x3f_process.c](src/x3f_process.c) in this order: black/white levels + spatial gain → color matrix + gamma LUT → bad-pixel interpolation → highlight recovery (chroma LUT, sat map, RepairPix, matrix-pathology gate) → the [convert_data hot loop](src/x3f_process.c#L1725). Replace the ~15 `X3F_*` env vars with a typed `ProcessingConfig` struct exposed via `clap` derive; `from_env()` reads the existing names for backwards compat with one deprecation cycle. Replace `*_TRACE` printfs with `tracing` spans. Highlight recovery is where the user's research lives — **port verbatim first, tidy in a separate PR**, and pair-review the highlight code.

- **M8d — WASM (wasm32-wasip1)** ([crates/x3f-sys/build.rs](crates/x3f-sys/build.rs), [crates/x3f-core/src/lib.rs](crates/x3f-core/src/lib.rs)) — *done.* `cargo build -p x3f-ffi-c --target wasm32-wasip1 --release` produces a 4 MB `x3f.wasm` cdylib + 17.7 MB `libx3f.a` staticlib + the same `x3f.h` cbindgen header used elsewhere; all 6 C ABI symbols (`x3f_reader_open` / `_close` / `_header_version`, `x3f_reader_dump_meta`, `x3f_reader_dump_jpeg_thumbnail`, `x3f_last_error`) export as wasm `command_export`s.

  Two infrastructure changes made this work:
  1. **`x3f-sys/build.rs` skips cc-rs entirely on `target.starts_with("wasm32")`** — Apple's bundled clang has no wasi-libc / wasm-libc sysroot, and the three remaining C files (`x3f_printf.c`, `x3f_version.c`, `csrc/denoise_stub.c`) all need `<stdio.h>` / `<inttypes.h>`. Bindgen still runs because the rest of the workspace consumes type definitions (`x3f_t`, `x3f_directory_entry_t`, etc.) from the generated `bindings.rs`; we override `--target=$HOST` for the bindgen clang invocation so it parses the headers against the host's `<inttypes.h>` (the integer widths are layout-compatible: `c_int` = i32 on both wasm32 and the build host). The blocklist application was extracted into a shared `apply_bindgen_blocklists` helper so the wasm32 and host paths use exactly the same symbol policy.
  2. **`x3f-core/src/lib.rs` cfg-gates `OsStrExt`** — `std::os::unix::ffi::OsStrExt` doesn't exist on `target_os = "wasi"`; the equivalent lives at `std::os::wasi::ffi::OsStrExt`. Both forms expose `as_bytes()` so `path_to_cstring` works unchanged.

  CI coverage in the `wasm32-wasip1` GitHub Actions job ([.github/workflows/ci.yml](.github/workflows/ci.yml)) on `ubuntu-latest` builds the crate and asserts every C ABI symbol is present in the wasm output, then uploads the artefacts.

  Bindgen's choice of clang `--target` is also *why* this works without shimming `x3f_printf` / `x3f_version` / `x3f_denoise` etc. into Rust — the C source isn't compiled, but the function declarations come from header parsing and bindgen marks them as extern; nothing in the x3f-ffi-c surface actually calls them at runtime, so dead-code elimination drops the unresolved imports before the linker would complain. If a future caller does pull one in (e.g. surfacing TIFF/DNG output to wasm), we'll need to ship matching Rust shim implementations under `cfg(target_arch = "wasm32")`.

- **M8d-α — WASM (wasm32-unknown-unknown)** ([crates/x3f-sys/src/sysabi.rs](crates/x3f-sys/src/sysabi.rs), [crates/x3f-sys/src/print_meta_wasm_stub.rs](crates/x3f-sys/src/print_meta_wasm_stub.rs), [crates/x3f-sys/src/histogram_wasm_stub.rs](crates/x3f-sys/src/histogram_wasm_stub.rs)) — *done.* `cargo build -p x3f-ffi-c --target wasm32-unknown-unknown --release` produces a 3.4 MB `x3f.wasm` cdylib + 17.5 MB `libx3f.a` staticlib + the cbindgen header; all 6 C ABI symbols export as wasm `command_export`s.

  The starting point was 561 compile errors — `wasm32-unknown-unknown`'s `libc` crate exposes a tiny sliver of the symbols x3f-sys uses (~589 `libc::*` references across the workspace). The fix is a `sysabi` compat module that's *transparent* on every other target (`pub use libc::*;`) and provides Rust-native equivalents on wasm32:

  | Category | wasm32-unknown-unknown source |
  |---|---|
  | Types (`c_int` / `c_char` / `c_void` / `c_long`) | `pub use core::ffi::*` |
  | Allocator (`malloc` / `calloc` / `realloc` / `free`) | `std::alloc::*` with size-prefix layout trick so `free` knows the size to deallocate |
  | Memory ops (`memcpy` / `memset`) | `core::ptr::copy_nonoverlapping` / `write_bytes` (compile to `memory.copy` / `memory.fill`) |
  | Conversion (`atof` / `atoi`) | `CStr → str::parse`, returning 0 on failure (matches C) |
  | File I/O (`fopen` / `fclose` / `fseek` / `ftell` / `fread` / `fgetc` / `fdopen`) | NULL-returning stubs — wasm32-unknown-unknown has no host filesystem; the buffer-based reader API tracked as M8d-α-2 replaces this surface |
  | Process exit (`abort` / `exit`) | `core::arch::wasm32::unreachable()` (a `unreachable` opcode) |

  Submodules opt into the shim with one line: `use crate::sysabi as libc;` (or `use x3f_sys::sysabi as libc;` from x3f-core). That shadows the external `libc` crate within the file so existing `libc::*` call sites compile unchanged on wasm32 — no body churn.

  Two surfaces were *not* shimmed and are gated out on wasm32 instead, because Rust can't define variadic functions on stable (`c_variadic` is unstable):
  1. [crates/x3f-sys/src/print_meta.rs](crates/x3f-sys/src/print_meta.rs) — the `-meta` text dump uses `libc::printf` / `fprintf` for byte-identical `%g` / `%9f` formatting (tier-2-MD5-pinned). Replaced on wasm32 by [crates/x3f-sys/src/print_meta_wasm_stub.rs](crates/x3f-sys/src/print_meta_wasm_stub.rs), which returns `X3F_ARGUMENT_ERROR`.
  2. [crates/x3f-sys/src/histogram.rs](crates/x3f-sys/src/histogram.rs) — the `-histogram` debug CSV writer. Replaced on wasm32 by [crates/x3f-sys/src/histogram_wasm_stub.rs](crates/x3f-sys/src/histogram_wasm_stub.rs).
  3. Three `X3F_CHROMA_LUT_*`-env-var-gated debug `fprintf` blocks in [crates/x3f-sys/src/highlight.rs](crates/x3f-sys/src/highlight.rs). `chroma_lut_apply_stats_print` itself becomes a no-op on wasm32; the inline `X3F_CHROMA_LUT_DUMP` block is `cfg(not(target_arch = "wasm32"))` directly.

  One additional cfg gate: `path_to_cstring` in [crates/x3f-core/src/lib.rs](crates/x3f-core/src/lib.rs) falls back to `OsStr::as_encoded_bytes()` on wasm32-unknown-unknown (where neither `std::os::unix` nor `std::os::wasi` modules exist).

  CI matrix in [.github/workflows/ci.yml](.github/workflows/ci.yml) tests both `wasm32-wasip1` and `wasm32-unknown-unknown` targets, asserts each of the 6 ABI symbols is present in the wasm output, and uploads the artefacts.

- **M8d-α-2 — Browser-runtime WASM (`Reader::from_bytes`)** ([crates/x3f-sys/src/sysabi.rs](crates/x3f-sys/src/sysabi.rs), [crates/x3f-sys/src/wasm_c_shims.rs](crates/x3f-sys/src/wasm_c_shims.rs), [crates/x3f-core/src/lib.rs](crates/x3f-core/src/lib.rs), [crates/x3f-ffi-c/src/lib.rs](crates/x3f-ffi-c/src/lib.rs)) — *done.* `Reader::from_bytes(&[u8])` parses an X3F file from an in-memory buffer. Backed by libc's `fmemopen` on Unix-likes, by an internal `MemFile` cursor in the wasm32 sysabi shim. The same Rust call sequence runs on host and on wasm — no API split.

  Implementation:
  1. **`MemFile`-backed `FILE*` in the wasm sysabi shim.** What was a NULL-returning stub in M8d-α now copies the input bytes into a heap-side cursor and returns a `Box::into_raw` pointer that aliases as `*mut FILE`. `fread` / `fseek` / `ftell` / `fgetc` / `fclose` walk the cursor; `fmemopen(buf, size, "rb")` is the new entry point. Each `MemFile` owns its bytes (we copy on `fmemopen`) so the `FILE*` outlives the caller's borrow — JS / Swift consumers don't have to keep the input pinned.
  2. **`#[no_mangle]` Rust shims for the four C symbols** ([crates/x3f-sys/src/wasm_c_shims.rs](crates/x3f-sys/src/wasm_c_shims.rs)) the bindgen output still references on wasm32: `x3f_printf`, `x3f_denoise`, `x3f_denoise_active`, `x3f_set_use_opencl`. All four are no-ops; without them the wasm cdylib emits `(import "env" "x3f_printf" ...)` entries that any consumer would have to satisfy. The variadic `x3f_printf` is shimmed with a 3-arg signature `(level, fmt, _arg)` — Rust on wasm32 lowers every variadic call site to import-type `(i32, i32, i32)` regardless of how many extras the C source declared, so a 3-arg impl satisfies them all. `#[used]` anchors in lib.rs prevent cross-crate dead-code elimination from stripping the symbols before the bindgen extern decls find them.
  3. **`Reader::from_bytes(&[u8])`** ([crates/x3f-core/src/lib.rs](crates/x3f-core/src/lib.rs)) — `cfg(any(unix, target_arch = "wasm32"))`-gated. Copies the input into a `Vec<u8>` stashed in `Reader._backing` so the FILE* cursor outlives any caller buffer, then calls `libc::fmemopen` (which dispatches through our shim on wasm32). Drop order is FILE* close → backing bytes free, matching the C parser's lifetime expectations.
  4. **`x3f_reader_open_from_bytes(data, len)`** in the C ABI. Returns NULL on Windows (no `fmemopen` in MSVC's CRT) with a `x3f_last_error` explanation; works everywhere else.

  **End-to-end runtime test.** [crates/x3f-core/examples/wasi_from_bytes.rs](crates/x3f-core/examples/wasi_from_bytes.rs) is a tiny wasi binary that reads an X3F file via `std::fs::read`, calls `Reader::from_bytes`, and prints the header version. [crates/x3f-core/tests/wasm_runtime_smoke.rs](crates/x3f-core/tests/wasm_runtime_smoke.rs) cross-compiles it for `wasm32-wasip1`, runs it under `wasmtime`, and asserts the wasm output matches the host `Reader::open` baseline byte-for-byte. Locally on a Quattro DP0Q file: `OK 0x00040001` matches host `Reader::open`'s `header_version()`. CI installs `wasmtime` via the official `wasmtime.dev/install.sh` script and runs the same test on `ubuntu-latest`.

  **`wasm32-unknown-unknown` invariant**: the cdylib has *zero* `(import "env" ...)` entries — fully self-contained. Validated in CI with `wasm-tools print | grep '(import "env"'`. That means a browser consumer can `WebAssembly.instantiateStreaming` the wasm with no host-function shims; the only imports are wasi (which `wasi_snapshot_preview1` polyfills handle for browser).

  **Test coverage.** Four host integration tests in [crates/x3f-core/tests/from_bytes.rs](crates/x3f-core/tests/from_bytes.rs): parity with `Reader::open` on header version, full metadata-dump pipeline through the buffer cursor, parse-error rejection of garbage / empty input. Plus the wasm runtime smoke above. [crates/x3f-ffi-c/tests/c_abi_smoke.rs](crates/x3f-ffi-c/tests/c_abi_smoke.rs) extended to reference `x3f_reader_open_from_bytes` so cbindgen drift on the new symbol surfaces immediately.

## Known regressions / follow-ups

- **DNG picture profiles not visible to downstream consumers (reported 2026-04-27).** The Rust DNG writer continues to emit all six camera profiles (Default + FOV Classic Blue + Grayscale + Grayscale red filter + Grayscale blue filter + Unconverted) — exiftool confirms the bytes are present in `IFD0` (Default) and five `ProfileIFD`s reached via `ExtraCameraProfiles`. User reports they no longer appear in the consumer used to inspect them (likely Lightroom's profile dropdown). Suspected cause: subtle structural difference in our hand-rolled MMCR mini-TIFFs vs libtiff's output (e.g. tag ordering inside the IFD body, missing per-profile `CalibrationIlluminant1`, missing `UniqueCameraModel`/`ProfileEmbedPolicy`/`ProfileCopyright`, or 2-byte alignment vs 4-byte). Investigation steps: (1) byte-diff a Rust-produced MMCR blob against a libtiff-produced one from a known-good legacy `x3f_extract` build for the same input; (2) try adding `CalibrationIlluminant1=D65` to each profile IFD; (3) test a known-good third-party DNG (Adobe DNG Converter output) and compare ExtraCameraProfiles structure. Marked **not critical** by user, but should land before crates.io publish (M9).
